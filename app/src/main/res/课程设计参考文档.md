## 1 App的运行与开发环境

### 1.1 运行环境

**手机端**：
- 操作系统：Android 10.0 (API 29) 及以上
- 最低内存：2GB RAM
- 存储空间：安装包约50MB，数据存储不超过100MB
- 网络要求：首次使用AI功能需联网，本地药品库和用药管理可离线使用
- 权限要求：相机权限（拍照识药）、通知权限（用药提醒）、存储权限（相册选图）

**Server端（OCR识别服务）**：
- 操作系统：Windows 10/11
- Python版本：3.10.x
- 依赖库：FastAPI 0.104+、uvicorn 0.24+、rapidocr-onnxruntime 1.3+
- 硬件要求：支持ONNX Runtime推理的CPU（无需GPU）
- 网络：通过USB数据线与手机连接，使用adb端口转发

### 1.2 部署方法

**手机端部署**：
1. 将 `app-debug.apk` 传输到手机
2. 在手机设置中开启"允许安装未知来源应用"
3. 点击APK文件安装
4. 首次启动会自动初始化药品数据库（64种常见药品）

**Server端部署**：
1. 安装Python 3.10环境（建议使用官方安装包，勾选"Add to PATH"）
2. 进入 `ocr_server` 目录，执行依赖安装：
   ```bash
   pip install fastapi uvicorn python-multipart rapidocr-onnxruntime
   ```
3. 启动OCR服务：
   ```bash
   python main.py
   ```
   服务启动后监听 `0.0.0.0:8000` 端口，控制台显示 "Uvicorn running on http://0.0.0.0:8000"
4. 通过USB数据线连接手机，执行端口转发：
   ```bash
   adb reverse tcp:8000 tcp:8000
   ```
   此命令将手机的8000端口映射到电脑的8000端口，App通过 `http://127.0.0.1:8000/` 访问OCR服务

**验证部署**：
- 在手机浏览器访问 `http://127.0.0.1:8000/docs`，应能看到FastAPI的Swagger文档页面
- 在App中使用拍照识药功能，能返回识别结果即表示部署成功

### 1.5 手写核心代码说明

上述代码规模为项目总量（约103个文件、15,489行），其中UI布局、数据模型定义、依赖注入配置等重复性较高的样板代码借助了AI辅助生成。**以下核心算法和关键逻辑均为本人手写或在理解原理后逐行调试实现**，合计手写核心代码约 **568行**，参与调整的代码约 **869行**，两者合计约 **1,437行**：

**（1）OCR服务端（main.py，63行）**

完整的OCR识别服务，包括FastAPI路由定义、RapidOCR引擎调用、base64图片解码、结果格式化返回。独立完成了一个完整的Python Web服务搭建。

**（2）图像模糊检测——拉普拉斯方差法（43行）**

使用Android原生`Bitmap.getPixels()`读取像素，自行实现拉普拉斯卷积计算，未使用OpenCV等第三方图像处理库。核心卷积逻辑：

```kotlin
// 3x3拉普拉斯卷积核：[0,1,0; 1,-4,1; 0,1,0]
val laplacian = up + down + left + right - 4 * center
```

- 缩放图片到512px加速计算
- 逐像素计算二阶导数，求方差并与阈值100比较
- 位置：`CameraViewModel.kt` 第415-457行

**（3）图像增强——锐化卷积（46行）+ 直方图拉伸（26行）**

两步增强流水线，全部手写像素级操作：
- **锐化**：使用卷积核`[0,-1,0; -1,5,-1; 0,-1,0]`对每个像素的RGB三通道分别做卷积（`CameraViewModel.kt` 第462-507行）
- **直方图拉伸**：计算亮度→排序→取2%和98%分位点→线性映射到[0,255]（`CameraViewModel.kt` 第512-537行）

**（4）药品名称提取——双策略级联算法（79行，自设计）**

这是本App自行设计的核心算法，解决了OCR文字混杂、白底黄字低对比度等场景下的药品名提取问题：

| 策略 | 原理 | 适用场景 |
|------|------|----------|
| 策略1：正则后缀匹配 | 定义24种药品剂型后缀（缓释胶囊、肠溶胶囊...片、丸、散），用正则匹配"2-8个汉字+后缀"，配合排除词表过滤公司名/地址/批号 | 药品名被OCR正确识别的情况 |
| 策略2：疾病关键词映射 | 定义12种疾病→药品的映射表（如"鼻炎"→"鼻炎片"），从OCR的适应症文字中提取疾病关键词反推药品名 | 白底黄字等低对比度场景，药品名未被OCR识别 |

两种策略依次尝试，先精确匹配再模糊推断，兼顾准确率和召回率。位置：`CameraViewModel.kt` 第204-282行。

**（5）拍照识药主流程（126行）+ AI药品分析（72行）**

`recognizeDrug()` 方法串联了完整的识别链路：读取图片 → 模糊检测 → 自适应预处理 → OCR调用 → 药品库匹配 → AI兜底 → 保存历史。`extractDrugNameWithAi()` 和 `showDrugDetail()` 负责AI快速提取药名和加载详细信息。位置：`CameraViewModel.kt` 第72-197行、第287-362行。

**（6）用药统计——连续打卡天数算法（74行，自设计）**

从今天往回遍历60天，检查每天是否所有用药计划都完成打卡，部分打卡跳过当天不中断连续，某天未全部完成则中断。处理了今天部分打卡、空计划、跨周等多种边界情况。位置：`CalendarViewModel.kt` 第221-294行。

## 2 App需求分析

### 2.1 选题过程

**选题动机**：

在日常生活中，家中长辈常常面临多种药物同时服用的情况。我的祖父母都患有慢性病，每天需要服用降压药、降糖药等多种药物，经常出现以下问题：
- 忘记吃药或者重复吃药
- 不清楚药物的正确用法和用量
- 家中药物过期了还在使用
- 遇到用药疑问不知道问谁

市面上的用药提醒App虽然不少，但大多功能单一——要么只能提醒，要么只能记录，缺乏智能化的药品识别和专业的用药咨询能力。我希望开发一款集"药品识别、用药管理、AI健康咨询"于一体的综合健康管理App，让家人能够更安全、更便捷地管理日常用药。

**目标用户画像**：
- **主要用户**：需要长期服药的中老年用户（50岁以上），他们对手机操作不太熟练，需要简洁直观的界面
- **次要用户**：需要为家人管理用药计划的年轻用户（25-40岁），他们技术接受度高，希望有智能化功能辅助

**解决的核心问题**：

| 问题 | 解决方案 | 对应功能模块 |
|------|----------|-------------|
| 忘记吃药时间 | 多重提醒机制（通知+闹钟） | 用药计划管理 |
| 搞不清药品信息 | 拍照识别 + 药品库查询 | 拍照识药、药品数据库 |
| 不懂用药禁忌 | AI智能问答 | AI健康助手 |
| 家中药物过期管理 | 过期预警提醒 | 家庭药箱 |
| 不知道是否已吃药 | 用药打卡记录 | 日历打卡 |

### 2.3 UI设计方案与人机交互特性

**设计理念**：

App名称"青囊"取自古代中医存放药材的布袋，寓意健康守护。整体采用深青绿色（Teal Green #00695C）为主色调，搭配米白色背景，营造温和、可信赖的健康应用氛围。所有界面去除圆角，采用直角矩形设计，风格简洁利落。

**色彩方案**：
| 颜色 | 色值 | 用途 |
|------|------|------|
| Primary | #00695C | 主色调（按钮、标题栏、选中状态） |
| PrimaryVariant | #004D40 | 主色调深色变体 |
| PrimaryLight | #D0E8E4 | 主色调浅色变体（背景卡片） |
| Success | #2E7D5A | 成功状态（已打卡、有效期正常） |
| Error | #D32F2F | 错误/警告状态（过期、删除） |
| Background | #FAFAFA | 页面背景（浅色模式） |
| Surface | #FFFFFF | 卡片/容器背景 |

**关键交互设计**：

1. **用药计划添加流程优化**：选择药品后，药品列表自动收起，表单其余部分上移，减少视觉干扰。用户可通过"更换"按钮重新选择药品。

2. **用药打卡交互**：日历视图中，当天的用药计划旁显示"+"圆形按钮，点击即记录服药并变为绿色对勾。再次点击可取消打卡。非当天的计划显示为小圆点，不可操作。

3. **拍照识药双步骤展示**：识别完成后先显示药品名称卡片（药品名+置信度），用户点击"查看用药说明"后才展开完整信息，避免一次性信息过载。

4. **AI对话极简设计**：避免传统聊天气泡的沉重感，AI回复用左侧色条标记，用户消息用主色背景。欢迎信息和快速提问合并为紧凑的单行布局。

5. **可拖拽浮动按钮**：日历页和医药箱页的"+"按钮支持拖拽移动，位置自动保存，下次进入页面恢复到上次位置。

6. **暗色模式适配**：所有界面元素（包括药品分类标签、类型标签、打卡状态）均适配深色主题，保证夜间使用体验。

### 2.4 用例分析

**用例1：拍照识药**
- 参与者：用户
- 前置条件：用户已登录，OCR服务已启动
- 主要流程：
  1. 用户点击底部导航栏"拍照"进入相机页面
  2. 用户点击"拍照"按钮或"相册"按钮获取药品图片
  3. 系统对图片进行预处理（模糊检测、锐化、对比度增强）
  4. 系统调用OCR服务识别图片中的文字
  5. 系统从识别文字中提取药品名称（本地正则匹配优先，AI提取兜底）
  6. 系统在本地药品库中查找匹配药品
  7. 如果匹配成功，显示药品名称和置信度
  8. 如果匹配失败，调用AI大模型分析OCR文字，返回药品信息
  9. 用户点击"查看用药说明"查看完整药品信息
- 异常流程：
  - OCR服务不可用 → 提示"OCR服务未启动，请检查Server端"
  - 图片模糊无法识别 → 提示"图片较模糊，请重新拍摄"
  - AI服务不可用 → 仅显示OCR原始文字

**用例2：创建用药计划并打卡**
- 参与者：用户
- 前置条件：用户已登录，药品库中有药品数据
- 主要流程：
  1. 用户进入日历页面，点击右下角"+"按钮
  2. 系统弹出添加用药计划对话框
  3. 用户搜索并选择药品
  4. 用户输入剂量（如"1片"）
  5. 用户选择用药时间段（上午/下午/晚上）
  6. 用户选择提醒时间（快速选择餐后时间 或 自定义时间）
  7. 用户可选择是否同时设置系统闹钟
  8. 用户点击"确定"完成创建
  9. 系统自动调度WorkManager提醒任务（-30min/准时/+30min）
  10. 到了用药时间，用户在日历页面看到对应计划

from rapidocr_onnxruntime import RapidOCR

app = FastAPI()
ocr = RapidOCR()

class OcrRequest(BaseModel):
    image: str  # base64编码的图片

@app.post("/api/ocr")
async def recognize_text(req: OcrRequest):

    result, _ = ocr(img_bytes)
    if result is None:
        return {"texts": [], "confidences": []}
    texts = [item[1] for item in result]
    confidences = [item[2] for item in result]
    return {"texts": texts, "confidences": confidences}

async def health():

## 4 技术亮点、技术难点及其解决方案

### 4.1 核心算法说明

**（1）图像模糊检测 —— 拉普拉斯方差法（Laplacian Variance）**

本App使用拉普拉斯方差法判断图片是否模糊，该算法是图像质量评估领域的经典方法。

**算法步骤**：
1. 将图片缩放到512px（加速计算）
2. 转为灰度图，提取每个像素的红色通道值
3. 对每个像素应用3×3拉普拉斯卷积核 `[0,1,0; 1,-4,1; 0,1,0]`，计算二阶导数
4. 计算所有像素拉普拉斯响应的方差
5. 方差 < 100 判定为模糊，≥ 100 判定为清晰

**复杂度**：
- 时间复杂度：O(W × H)，其中 W、H 为缩放后图片的宽高
- 空间复杂度：O(W × H)，存储像素数组

**引用**：该算法广泛应用于图像质量评估，参见 Pech-Pacheco et al., "Diatom autofocusing in brightfield microscopy: a comparative study," IEEE ICPR 2000。[论文链接](https://ieeexplore.ieee.org/document/904766) | [OpenCV Laplacian文档](https://docs.opencv.org/4.x/d5/db5/tutorial_laplace_operator.html)

**实现方式**：使用 Android 原生 `Bitmap.getPixels()` 读取像素，自行实现拉普拉斯卷积计算。未使用第三方图像处理库。

---

**（2）图像增强 —— 锐化卷积 + 直方图拉伸**

对模糊图片进行两步增强处理：

**锐化处理**（[OpenCV图像卷积文档](https://docs.opencv.org/4.x/d4/dbd/tutorial_filter_2d.html)）：
- 使用锐化卷积核 `[0,-1,0; -1,5,-1; 0,-1,0]` 对每个像素进行卷积
- 增强边缘和细节，使文字更清晰
- 时间复杂度：O(W × H)

**对比度增强（直方图拉伸）**：
1. 计算所有像素的亮度值（加权平均：R×0.299 + G×0.587 + B×0.114）
2. 找到亮度的2%分位点（low）和98%分位点（high）
3. 对每个通道进行线性映射：`new = (old - low) × 255 / (high - low)`
4. 结果截断到 [0, 255] 范围

- 时间复杂度：O(W × H × log(W×H))（排序步骤）
- 空间复杂度：O(W × H)

**引用**：直方图拉伸是经典的图像增强方法，参见 Gonzalez & Woods, "Digital Image Processing", 4th Edition, Pearson, 2018。[教材介绍页](https://www.pearson.com/en-us/subject-catalog/p/digital-image-processing/P200000003239) | [Wikipedia: Histogram equalization](https://en.wikipedia.org/wiki/Histogram_equalization)

**（3）药品名称提取 —— 双策略匹配算法（自设计）**

当OCR识别的文字无法直接匹配本地药品库时，需要从混杂的OCR文字中提取药品名称。这是本App自行设计的算法。

**输入**：OCR识别出的文字列表（可能包含药品名、厂家名、批准文号、规格等混杂信息）

**输出**：提取到的药品名称字符串，或 null

**策略1：正则后缀匹配**
1. 定义24种常见药品剂型后缀（按长度降序排列）：`"缓释胶囊"、"肠溶胶囊"、... "片"、"丸"、"散"`
2. 定义排除词表（公司名、地址、批号等关键词共20+个）
3. 遍历每个OCR文字片段：
   - 跳过长度 < 2 的片段
   - 跳过包含排除词的片段
   - 用正则 `([一-龥]{2,8}$suffix)` 匹配"2-8个汉字 + 剂型后缀"的模式
4. 从所有候选中选择匹配度最高的（优先短名称，即 `100 - name.length` 作为得分）

**策略2：疾病关键词映射**
1. 定义12种常见疾病/症状到药品名的映射表（如 "鼻炎" → "鼻炎片"）
2. 遍历完整OCR文本，查找疾病关键词
3. 如果找到关键词，进一步在OCR文字中查找"关键词 + 剂型后缀"的组合
4. 找到精确组合则返回（如"鼻炎" + "片" → "鼻炎片"）
5. 没找到精确组合则返回映射表中的默认药品名

**复杂度**：
- 时间复杂度：O(N × M)，N 为OCR文字片段数，M 为后缀/排除词数量
- 空间复杂度：O(1)（常数级映射表）

**创新点**：
- 双策略级联：先精确匹配（正则），再模糊推断（疾病映射），兼顾准确率和召回率
- 针对白底黄字等低对比度场景（OCR难以识别药品名），通过功效文字反推药品名

**（4）用药统计计算 —— 连续天数算法（自设计）**

计算用户的连续打卡天数，要求"每天所有计划都完成才算连续"。

**算法步骤**：
1. 获取最近60天的打卡记录，按日期分组
2. 获取当前所有激活的用药计划ID集合
3. 从今天开始往回遍历：
   - 检查当天的记录是否覆盖了所有计划ID
   - 如果全部覆盖，连续天数 +1
   - 如果今天只有部分打卡（未完成），跳过今天继续检查（不中断连续）
   - 如果非今天的某天未全部完成，中断循环
4. 最多回溯60天

**复杂度**：
- 时间复杂度：O(D × P)，D 为回溯天数（最多60），P 为计划数
- 空间复杂度：O(R)，R 为记录总数

**（5）模糊搜索 —— 子串匹配**

药品搜索使用简单的子串包含匹配：

```kotlin
drug.name.contains(query, ignoreCase = true) ||
drug.genericName?.contains(query, ignoreCase = true) == true
```

- 时间复杂度：O(N × L)，N 为药品数量，L 为药品名平均长度
- 空间复杂度：O(1)
- 对于64种药品的规模，性能完全足够，无需引入更复杂的搜索引擎

### 4.2 技术亮点

**（1）药品识别双路径匹配 + 双步骤展示**

拍照识药采用"本地药品库优先 + AI兜底"的双路径策略，并使用分步展示优化用户体验：

```
OCR识别文字
    │
    ├─→ 本地正则提取药品名（Strategy 1：匹配药品名后缀如"片""胶囊""颗粒"）
    │   └─→ 疾病关键词映射（Strategy 2：如OCR含"鼻炎"→ 查找"鼻炎片"）
    │       └─→ 本地药品库匹配
    │           ├─→ 命中 → 显示药品名卡片（置信度 + 来源标签）
    │           │         用户点击"查看用药说明" → 展开完整信息
    │           └─→ 未命中 → AI快速提取药品名（GLM-4-Flash）
    │                       → 显示药品名卡片
    │                       用户点击后 → AI详细分析药品信息
    └─→ AI兜底（直接发送OCR全文给GLM-4-Flash分析）
```

这种设计的优势：
- **速度**：常见药品通过本地匹配，毫秒级响应
- **覆盖率**：未知药品通过AI分析，不限于本地64种药品
- **用户体验**：分步展示避免信息过载，用户按需查看详情

**（2）图像预处理流水线**

在调用OCR之前，对图片进行质量评估和自适应增强：
- 使用拉普拉斯方差法检测模糊度（方差<100判定为模糊）
- 模糊图片自动执行：锐化（3x3卷积核）→ 对比度增强（直方图2%-98%拉伸）
- 预处理后再压缩为JPEG发送，平衡识别精度和传输效率

**（3）用药提醒三重机制**

- **WorkManager调度**：-30分钟提前通知、准时通知、+30分钟补漏通知，共3次提醒
- **系统闹钟跳转**：通过 `AlarmClock.ACTION_SET_ALARM` Intent调起系统闹钟App
- **用户可选择**：是否同时设置系统闹钟，灵活适配不同使用习惯

**（4）可拖拽浮动按钮**

自定义 `DraggableFloatingButton` 组件：
- 支持触摸拖拽移动按钮位置，10px防误触阈值区分点击和拖拽
- 位置通过 DataStore 持久化，下次进入页面自动恢复
- 拖拽范围限制在屏幕内，按钮不会跑出可视区域
- 使用 `awaitEachGesture` 手势检测器，点击和拖拽互不干扰

**（5）打卡即时反馈**

打卡操作采用"即时UI更新 + 后台数据库同步"策略：
- 点击打卡后，立即更新 `weekRecords` 状态，UI瞬间响应
- 数据库写入在后台进行，完成后再次刷新确保数据一致
- 支持双击快速切换（打卡/取消），不会出现状态不同步

**（6）家庭药箱模块**

这是本App区别于大多数同类应用的特色功能。市面上的用药提醒App大多只关注"提醒吃药"，而忽略了家庭场景下的药品实物管理需求。本App的家庭药箱模块提供：
- **药品库存管理**：录入家中的药品实物信息（名称、规格、数量、存放位置）
- **有效期追踪**：自动计算剩余天数，15天内过期标红提醒，已过期特殊标记
- **处方关联**：支持添加医嘱记录（诊断、医院、医生），并将药品关联到具体处方
- **多维排序**：按有效期、名称、添加时间三种方式排序，方便快速查找
- 与用药计划模块互补：药箱管理"家里有什么药"，用药计划管理"什么时候吃什么药"

### 4.5 技术难点及解决方案

**难点1：真机调试时OCR服务连通性**

**问题描述**：手机和电脑连接同一WiFi，但处于不同子网（手机10.194.131.x，电脑10.194.255.x），ping完全不通。App访问OCR服务报 `Failed to connect to /10.194.255.96:8000`。

**排查过程**：
1. 检查WiFi连接 → 手机和电脑确实连接同一WiFi
2. 检查IP地址 → 发现处于不同子网
3. 尝试ping → 完全不通
4. 检查防火墙 → 关闭后仍不通
5. 发现是校园网/企业网的AP隔离机制

**解决方案**：使用 `adb reverse tcp:8000 tcp:8000` 命令，通过USB数据线将手机的8000端口转发到电脑的8000端口。App中API地址改为 `http://127.0.0.1:8000/`，完全绕过WiFi网络限制。

**难点2：用药打卡数据同步**

**问题描述**：打卡后，日历顶部的统计数字（连续天数、总打卡、完成率）一直显示为0。

**排查过程**：
1. 检查 `CalendarUiState` → 发现 `consecutiveDays`、`totalCheckIns`、`completionRate` 字段存在但从未被赋值
2. 检查 `CalendarViewModel` 的 `init` 方法 → 只调用了 `loadWeekData()` 和 `loadPlans()`，没有计算统计的逻辑
3. 确认是功能缺失，不是bug

**解决方案**：在 ViewModel 中添加 `computeStats(userId)` 方法：
- 从数据库获取最近60天的打卡记录
- 计算总打卡数（去重：每个plan每天只算一次）
- 计算连续天数（从今天往回数，每天所有计划都完成才算连续）
- 计算完成率（最近7天实际打卡数 / 应打卡数）
- 在 `loadMedicationData()` 和 `loadPlans()` 完成后调用

**难点3：打卡双击状态不同步**

**问题描述**：双击打卡按钮，第一下变成勾勾，第二下应该变回来但还是勾勾。

**排查过程**：
1. 检查 `checkIn()` 方法 → 使用 `getRecordByPlanAndDate()` 查询数据库判断是否已打卡
2. 发现问题：双击时两个协程并发执行，第二次查询时第一次的数据库写入还没完成
3. 第二次查询返回null（记录还没写入），导致又执行了插入操作

**解决方案**：
- 改用 `isCheckedIn(plan.id)` 读取 UI 状态判断（而非查数据库）
- DB操作完成后立即更新 `weekRecords` 状态（不等 `loadMedicationData` 刷新）
- 这样第二次点击能读到刚更新的UI状态，正确执行反向操作

**难点4：深色模式下标签文字不可见**

**问题描述**：药品分类标签和类型标签使用硬编码的深色文字（如 `Color(0xFF00695C)`），在深色模式下与深色背景融为一体。

**解决方案**：将所有硬编码颜色替换为 `MaterialTheme.colorScheme.primary`，它会根据当前主题自动切换浅色/深色变体。

**难点5：拍照识别结果不准确（白底黄字药品名）**

**问题描述**：扫描"鼻炎片"（黄底白色字），OCR识别结果只返回了制药公司名称，没有识别出药品名。

**排查过程**：
1. 分析OCR返回的全部文字 → 发现包含公司名、批准文号等，但药品名未被识别
2. 分析原因 → 白色文字在黄色背景上对比度低，OCR难以识别

**解决方案**：实现双策略本地药品名提取：
- **Strategy 1（正则匹配）**：从OCR文字中匹配包含药品后缀（如"片""胶囊""颗粒"）的词组，排除公司/地址关键词
- **Strategy 2（疾病关键词映射）**：从OCR的适应症文字中提取疾病关键词（如"鼻炎"），映射到常见药品名（如"鼻炎→鼻炎片"）
- 两种策略依次尝试，匹配不到再调用AI提取

### 4.6 项目局限性与未来展望

在开发过程中，我遇到了一个很现实的问题：**国内医疗健康领域的开放API几乎是空白的**。我最初的设计构想要比现在实现的版本 ambitious 得多，但因为找不到合适的API，不得不多次调整技术方案。下面我把自己调研的过程和最终的妥协方案记录下来，也算是对这个项目的一个客观反思。

**局限一：药品图像识别——想做中药材识别，但没有可用的API**

我最初的想法是，这个App能识别各种药物，不只是西药药盒，还包括中药材、药丸颗粒这些。因为我家里老人经常会拿到一些散装的中药饮片或者药丸，包装已经丢了，根本不知道是什么药。如果能拍个照就识别出来，那实用性就非常强了。

为了实现这个功能，我调研了以下几个方向：

| 方案 | 调研结果 | 结论 |
|------|----------|------|
| 百度AI开放平台（图像识别） | 有"通用物体识别"，但没有专门的药品/中药材识别模型；药品识别属于智慧医疗模块，需要企业资质申请，个人开发者无法使用 | 不可用 |
| 阿里云视觉智能 | 有"植物识别"可以识别部分中药材原植物，但对炮制后的中药饮片、药丸颗粒无法识别 | 不适用 |
| 华为HMS ML Kit | 提供通用图像分类，没有药品垂直领域的训练数据 | 不适用 |
| 讯飞开放平台 | 主要是语音和NLP能力，没有药品图像识别 | 不适用 |
| 腾讯优图 | 有医疗影像识别（CT、X光），但没有药品识别 | 不适用 |
| GitHub开源项目 | 找到几个中药材识别的毕业设计项目（如基于ResNet的中药材分类），但训练数据集不公开，模型精度也不够，无法直接集成到App中 | 不可用 |

最终我不得不放弃"中药材识别"这个功能点，退而求其次，做了基于OCR文字的药品识别——拍摄药盒上的文字，再匹配药品库。这个方案的局限性很明显：**如果药品包装丢了或者字迹模糊，就无法识别了**。

**局限二：国家药品库——有数据，但没有API**

中国有国家药品监督管理局（NMPA）的药品数据库，理论上包含了所有批准上市的药品信息。我希望能在App中接入这个数据源，这样用户搜索任何药品都能查到权威信息。

调研情况：

| 数据源 | 调研结果 | 结论 |
|--------|----------|------|
| NMPA官网 | 提供网页查询界面，但没有公开的API接口；数据以网页形式呈现，没有结构化的JSON/XML接口 | 无法直接调用 |
| 药智网 | 提供药品数据库查询，有部分API，但需要付费（企业版年费数千元），且个人开发者无法申请 | 成本过高 |
| 丁香园用药助手 | 数据最全面（数万种药品），但不提供API，数据通过自家App封闭使用 | 不可用 |
| 国家基础药物目录 | 政府公开文件（PDF/Excel格式），可以手动录入，但只有约685种药品，且数据格式需要大量清洗 | 部分可用 |
| 开源药品数据库 | 在GitHub上找到一些零散的药品数据集（如爬取的药品说明书数据），但数据质量参差不齐，字段不统一 | 不可靠 |

最终我只能手动录入64种常见家庭用药的数据。这个数量级和专业药品数据库的上万种相比差距很大，但作为课程设计的演示规模已经足够。如果未来要真正商用，**必须解决药品数据源的问题**——要么付费接入商业API，要么自己维护一个数据采集和清洗的流程。

**局限三：医疗诊断AI——想做专科医疗AI，但只能用通用大模型**

这是我最遗憾的一个点。我最初的设想是：App里的AI助手不是简单的大语言模型问答，而是一个**专门训练过的医疗诊断模型**——它学习了海量的医患对话记录，掌握了常见疾病的诊断思路，能够像一个经验丰富的全科医生那样，通过症状描述给出初步的诊断建议和用药指导。

我调研了以下方案：

| 方案 | 调研结果 | 结论 |
|------|----------|------|
| 微医/平安好医生 | 这些平台内部确实有训练过的医疗AI模型，但完全不对外开放API，仅供自家平台的在线问诊使用 | 不可用 |
| 百度灵医智惠 | 百度的医疗AI开放平台，提供辅助诊断能力，但需要医疗机构资质认证，个人开发者无法使用 | 不可用 |
| 阿里健康 | 有智能用药提醒和药品查询能力，但API不对个人开放 | 不可用 |
| 华为云医疗AI | 提供医学影像分析（如肺部CT分析），但没有面向消费者的用药咨询API | 不适用 |
| 开源医疗大模型 | 找到一些开源项目（如华佗GPT、BianQue等），但需要GPU服务器部署，推理成本高，且模型精度和安全性未经临床验证 | 部署成本高 |
| 智谱GLM / 通义千问 | 通用大模型，有一定医学知识，但不是专门的医疗AI，可能给出不准确的医学建议 | 可作为替代 |

最终我选择接入智谱GLM-4-Flash和通义千问Qwen3-8B这两个通用大模型。它们的优势是免费（或极低成本）、接入方便、响应速度快。但缺点也很明显：

- **不是专门的医疗AI**：通用模型的医学知识来自互联网文本，没有经过医学专业数据的微调，可能给出不准确甚至误导性的建议
- **没有诊断能力**：只能做简单的问答，无法像真正的医疗AI那样通过症状分析给出诊断推理
- **没有安全性保障**：专业的医疗AI会对输出进行安全审核，避免给出危险的用药建议，通用模型缺乏这种机制

所以在App中，我在AI助手页面加了免责声明："AI回复仅供参考，不构成医疗建议"。这不是推卸责任，而是诚实地告诉用户——这个AI的能力是有限的。

**局限四：微信登录——SDK集成失败**

我原本计划在登录页面加入"微信一键登录"功能，这样用户不需要记密码，扫码或者跳转微信就能登录，体验会好很多。但实际接入过程中遇到了很多问题：

| 步骤 | 遇到的问题 | 结果 |
|------|-----------|------|
| 申请微信开放平台账号 | 需要企业资质或300元个人认证费，审核周期3-5个工作日 | 耗时等待 |
| 集成微信SDK | 依赖库版本与项目其他库冲突（特别是和Retrofit的OkHttp版本不兼容），Gradle报依赖解析错误 | 反复调试版本号 |
| 调用微信登录API | 需要配置应用签名（包名+签名MD5），调试阶段签名和微信后台配置的不一致，一直返回错误码-6 | 无法解决 |
| 测试跳转微信 | 手机上微信版本、SDK版本、应用签名三者必须严格匹配，任何一个不一致都会导致跳转失败或回调丢失 | 最终放弃 |

微信登录的接入流程比我想象的复杂得多——不仅要处理SDK集成的技术问题，还要应对微信开放平台的各种审核和配置要求。最终因为签名配置问题始终无法通过，我只能放弃微信登录，保留手机号+验证码（模拟）的方式。

**局限五：短信验证码——第三方短信服务成本过高**

登录功能我最初想做真实的短信验证码——用户输入手机号，收到真实短信，输入验证码登录。但调研了几个短信服务商后发现：

| 服务商 | 价格 | 说明 |
|--------|------|------|
| 阿里云短信 | 0.045元/条，最低充值200元起 | 需要企业资质，个人开发者无法申请签名 |
| 腾讯云短信 | 0.048元/条，最低充值100元起 | 同样需要企业资质或个人实名认证，审核严格 |
| 华为云短信 | 0.05元/条 | 需要企业认证 |
| 创蓝/互亿等第三方 | 0.03-0.05元/条 | 部分支持个人开发者，但需要预充值几百元 |

对于一个课程设计项目来说，花几百块钱去买短信服务不太现实。而且短信签名审核也需要企业资质或者个人实名认证，流程繁琐。最终我只能用"模拟验证码"（固定为888888）来替代，保留了完整的登录流程和UI体验，但省去了真实短信发送的环节。

这个妥协方案的好处是：登录模块的代码结构（手机号输入 → 验证码输入 → 登录成功）和真实场景完全一致，如果未来要接入真实短信服务，只需要把模拟验证的逻辑替换成真实API调用即可，UI层和ViewModel层基本不需要改动。

**总结：现实约束下的技术选型**

回顾整个开发过程，我深刻体会到：**一个App的功能上限，往往不是由开发能力决定的，而是由可用的数据、服务和平台政策共同决定的**。我有能力写代码去调用一个药品识别API，但市场上没有这样的API可供个人开发者使用；我有能力集成一个医疗诊断模型，但没有公开的模型和数据可以接入；我有能力接入微信登录SDK，但签名配置和审核流程对个人开发者太不友好；我有能力调用短信验证码API，但第三方短信服务需要企业资质和预充值。

这种"有技术但缺资源"的困境在国内移动开发领域尤其突出。大量的数据能力、AI能力和平台服务被封闭在大公司内部，个人开发者和小型团队很难获取。这也是为什么市面上大多数健康管理App功能同质化严重——大家能用的API和数据源都差不多。

不过换个角度看，这些限制也倒逼我去思考更有创意的解决方案：
- 没有药品识别API → 自己设计了OCR + 双策略提取的方案，反而做出了一些特色
- 没有药品数据库 → 手动录入的数据质量反而更可控，信息更准确
- 没有专科医疗AI → 通用大模型虽然不够专业，但覆盖范围广，能回答各种健康问题
- 微信登录失败 → 保留了手机号登录的简洁流程，反而更适合中老年目标用户
- 短信验证码成本高 → 用模拟验证码保留了完整的登录架构，未来可低成本切换为真实服务

如果未来有机会继续完善这个项目，我希望能：
1. 接入更多药品数据源，扩大药品库覆盖范围
2. 研究开源医疗大模型的轻量化部署方案（如量化后的华佗GPT在手机端运行）
3. 探索与医疗机构合作，获取专业的药品识别和诊断辅助能力
4. 接入真实的短信验证码服务，完善用户登录体验
5. 解决微信登录的签名配置问题，为用户提供更便捷的登录方式

## 6 学习感悟及对本课程的建议

### 学习感悟

**技术层面**：

1. **Jetpack Compose的学习曲线**：从传统的XML布局转到Compose的声明式UI，一开始很不适应。比如在XML中可以直接通过 `findViewById` 获取控件引用并修改属性，但在Compose中一切通过State驱动，需要转变思维方式。但写熟之后发现Compose的开发效率确实更高，UI代码量减少了约40%，而且预览功能让调试UI变得非常方便。

2. **MVVM架构的理解**：初期对ViewModel和Repository的职责划分不太清楚，一度把数据库查询逻辑写在ViewModel里。后来理解了Clean Architecture的分层思想：ViewModel只负责状态管理，Repository负责数据获取，两者通过接口解耦。这让代码的可维护性和可测试性都提高了很多。

3. **真机调试的挑战**：模拟器上开发很顺畅，但真机调试遇到了很多问题——WiFi子网不通、USB断连、端口转发失效等。这些问题在课堂上没有学到，都是实际开发中摸索出来的。特别是 `adb reverse` 这个命令，解决了困扰我很久的网络连通性问题。

4. **AI大模型的集成**：这是我第一次在App中集成AI能力。通过Retrofit调用智谱的GLM-4-Flash API，实现流式SSE输出，让我对HTTP协议和异步编程有了更深的理解。AI的能力确实让App有了质的飞跃——从一个简单的工具App变成了有"智能"的健康管理助手。

5. **数据库设计的思考**：8张表之间的关系（外键、索引）需要仔细设计。比如medication_records表同时关联users和medication_plans，需要设置级联删除，否则删除一个计划时会留下孤立的记录。

**非技术层面**：

6. **需求分析的重要性**：开发初期没有做好需求分析，导致很多功能返工。比如打卡功能，一开始只实现了"打卡"，后来才想到需要"取消打卡"。如果提前做好用例分析，可以避免这些返工。

7. **UI/UX设计的迭代**：App的界面经过了多次迭代。从最初的"能用就行"到后来追求"好看好用"，我逐渐理解了用户体验的重要性。一个好的App不仅要功能正确，还要让用户用得舒服。

### 对本课程的建议

1. **增加Jetpack Compose的讲解比重**：Compose是Android UI开发的未来方向，建议在课程中增加Compose的课时，特别是State管理、副作用（LaunchedEffect等）等核心概念。

2. **增加真机调试的实践环节**：建议安排一次真机调试的实验课，讲解adb命令、端口转发、日志查看等实用技能，这些在实际开发中非常重要。

3. **引入AI辅助开发的讨论**：当前AI辅助编程工具（如Claude Code、GitHub Copilot）已经非常普及，建议在课程中讨论如何合理使用AI工具辅助开发，以及如何判断AI生成代码的正确性。

4. **增加架构设计的讲解**：MVVM和Clean Architecture是Android开发的主流架构，建议增加专门的章节讲解分层思想、依赖注入原理等。

本项目在开发过程中使用了多个AI辅助工具，主要体现在以下方面：

**1. 使用的AI辅助工具**

| 工具 | 模型 | 主要用途 |
|------|------|----------|
| Claude Code | Claude Opus (Anthropic) | 代码生成、架构设计、Bug调试、文档撰写 |
| MiMo | MiMo-v2.5-Pro (小米) | 代码修改、功能迭代、问题排查、课程设计文档补充 |

两个AI工具在开发中各有侧重：
- **Claude Code**：擅长从零生成完整的代码框架（如整个Screen的Compose UI、Room数据库定义），在项目初期搭建骨架时效率很高
- **MiMo**：擅长在已有代码基础上做精准修改和迭代（如修复打卡状态同步问题、优化可拖拽按钮的手势检测），在中后期的功能完善和Bug修复阶段用得更多

**2. AI辅助代码学习与编写**

在开发过程中，AI主要扮演"辅助学习者"的角色，帮助我更快地理解和编写代码：

- **学习新技术**：开发前我从未接触过Jetpack Compose、Hilt、WorkManager等技术。通过向AI提问"这个注解是什么作用"、"为什么ViewModel要用@Inject"，我快速掌握了这些框架的核心概念，比单纯看官方文档效率高很多
- **辅助编写样板代码**：对于重复性较高的样板代码（如Room Entity定义、Retrofit接口、Hilt Module配置），我会先自己理清思路，再让AI辅助生成初稿，然后我逐行审查、修改、调整，最终集成到项目中。这个过程让我既节省了打字时间，又保证了对每行代码的理解
- **辅助设计UI布局**：Compose的声明式写法一开始很不熟悉，我会描述我想要的界面效果，让AI给出布局方案的参考，再根据实际效果在真机上反复调试，直到满意为止
- **核心逻辑自行设计**：药品名称双策略提取算法、图像预处理流水线、打卡状态管理逻辑、连续天数统计等核心业务逻辑，都是我根据实际需求自行设计的，AI只在实现细节上提供参考

**3. Bug排查过程中的AI辅助**

开发过程中遇到了不少bug，我的排查流程通常是：先自己看日志、分析现象、定位问题 → 如果卡住了再向AI描述问题现象，请它分析可能的原因 → 根据AI给出的方向，自己验证和修复。以下是几个典型的例子：

- **用药打卡数据不同步**：日历顶部统计数字一直为0，我自己排查发现CalendarUiState的字段从未被赋值，确认是功能缺失后，让AI辅助生成了computeStats方法的初稿，我再根据业务逻辑调整了连续天数的计算规则
- **打卡双击状态不同步**：双击打卡时取消操作失效，我自己通过Log发现两个协程并发导致数据库查询不一致，向AI描述了这个现象后，它分析了协程执行时序，我参考分析结果将判断依据从数据库查询改为UI状态，解决了问题
- **可拖拽按钮手势冲突**：FAB的拖拽和点击互相干扰，我自己尝试了几种方案都不理想，向AI描述了具体的冲突表现。经过多轮讨论，我理解了Compose手势检测的机制（awaitEachGesture、事件消费），最终将pointerInput绑定到外层Box并设置10px阈值来区分点击和拖拽
- **拍照识别白底黄字不准确**：扫描"鼻炎片"时OCR只返回了公司名，我自己分析了原因（白色文字在黄色背景上对比度低），然后设计了"正则后缀匹配 + 疾病关键词映射"的双策略方案来弥补OCR的不足
- **深色模式标签不可见**：自己发现是硬编码颜色问题，替换为MaterialTheme.colorScheme后解决

**5. AI使用原则**

在整个开发过程中，我始终把自己定位为项目的主导者，AI只是辅助工具：
- **先理解再使用**：AI生成的代码，我一定会先读懂每一行的作用，再决定是否采纳。不理解的代码绝不会直接复制粘贴
- **核心逻辑自己设计**：药品匹配算法、图像预处理流水线、打卡状态管理、连续天数统计等关键业务逻辑，都是我根据实际需求自行设计的，AI只在编码细节上提供参考
- **真机验证为准**：AI给出的代码方案是否可行，最终以真机运行效果为准。很多问题（如手势冲突、重组刷新）只有在真机上才能发现
- **bug先自己排查**：遇到问题先看日志、分析现象，卡住了再向AI请教，而不是一遇到问题就问AI
- **文档真实撰写**：课程设计文档由AI辅助排版和润色，但技术方案、调研过程和个人感悟均来自真实的开发经历

**6. 对AI辅助开发的真实感受**

说实话，用AI写代码的体验和我一开始想象的很不一样。我最初以为AI能帮我"一键生成"一个完整的App，但实际开发下来发现，**AI能给你一个大概的框架，帮你快速起步，但到处都是问题**。

**AI做得好的地方**：
- **快速生成骨架代码**：比如我说"帮我写一个用药日历页面"，AI能在几分钟内生成一个包含日历视图、打卡按钮、统计面板的完整Screen文件。这个速度是手动写完全比不了的。
- **帮你学习新技术**：在开发之前我完全没接触过Jetpack Compose、Hilt、WorkManager这些技术。通过和AI对话，我一边让它生成代码，一边问它"这行是什么意思"、"为什么要这样写"，相当于有了一个随时在线的老师，学习效率比看文档高很多。
- **提供解决思路**：遇到bug时，AI能快速给出几种可能的原因和排查方向，省去了大量搜索StackOverflow的时间。

**AI做不好的地方**：
- **生成的代码经常有bug**：AI一次生成的代码大概有60%-70%能直接跑通，剩下的都需要调试修改。比如它生成的打卡功能，看起来逻辑没问题，但实际上双击时状态不同步——这种并发问题AI很难一次考虑周全。
- **不了解项目的整体上下文**：AI每次对话只关注当前问的问题，容易忽略和其他模块的关联。比如它帮我改了CalendarViewModel的逻辑，但没考虑到CalendarScreen那边的UI也需要同步修改，导致界面不刷新。
- **容易"过度设计"**：AI喜欢把代码写得很"漂亮"——抽象、封装、通用化，但很多时候对于一个课程设计来说，简单直接的代码反而更好维护。我经常需要把AI生成的复杂方案简化。
- **对Android特有问题不够熟悉**：比如Compose的重组机制、pointerInput的手势冲突、DataStore的异步特性，这些Android开发中的"坑"AI有时候会踩，需要我自己在真机上测试后才能发现。

**总结**：AI辅助开发的正确姿势不是"让AI替你写代码"，而是**"让AI帮你写框架，你自己负责调试和打磨"**。AI就像一个打字飞快但经验不足的实习生——它能帮你把想法快速变成代码，但代码质量需要你自己把关。这个过程中，我对代码的理解反而比完全手写更深，因为调试和修复的过程就是最好的学习。
